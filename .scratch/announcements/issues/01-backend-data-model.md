# 01 — Backend: announcements table, model, and DAO

**What to build:** The persistence layer for announcements: the SQLAlchemy model, the Alembic migration, and a DAO module with the transactional single-active-invariant logic. No HTTP endpoints yet — that's ticket 02.

**Blocked by:** None — can start immediately

**Status:** ready-for-agent

- [x] `Announcement` model added to `backend/src/backend/db/models.py`: `id`, `title` (`String`, max 80 enforced at the Pydantic layer not DB), `body` (`String`, max 500), `link_label` (`String`, nullable), `link_url` (`String`, nullable), `is_active` (`Boolean`, default `True`), `created_at` (`DateTime(timezone=True)`, `server_default=func.now()`), `created_by` (`String`, nullable, mirrors `BronzeSnapshot.uploaded_by`) — table `announcements` in the `kiittime` schema, same pattern as `BronzeSnapshot`
- [x] Alembic migration `a1c3e9f0d2b4_add_announcements_table.py` added under `backend/src/backend/db/migrations/versions/` creating the table (follows the shape of `301dc7c59778_add_bronze_snapshots_table.py`)
- [x] `backend/src/backend/api/dao/announcement_dao.py` exports `create_announcement`/`get_current_announcement`/`deactivate_current` (placed under `api/dao/`, not `db/dao/`, matching the existing `section_dao.py`/`timetable_dao.py` location — no `db/dao/` directory exists in this codebase); `create_announcement` deactivates all prior rows then inserts+flushes the new one, no internal commit, matching `gold_upsert`/`process_upload`'s convention
- [x] DAO-level tests written in `backend/tests/test_announcements.py` against the real-Postgres `db` fixture (lifted into a new root `backend/tests/conftest.py` since the existing fixture only lived under `tests/pipeline/conftest.py`), covering: second announcement deactivates the first; current-announcement lookup returns most recent active row or `None`; `deactivate_current` clears without inserting and is a no-op when nothing is active — **verified**: `uv run pytest tests/test_announcements.py -v` passes locally against the migrated DB (see the `kiittime_app` role rename note below), full suite also green.
- Note: local Postgres role was renamed from `kiittime` to `kiittime_app` (see `backend/.env`) to fix a false-positive Alembic autogenerate diff caused by `POSTGRES_USER` colliding with the `kiittime` schema name (Postgres's `$user` search_path default). Migration `47595667330f_add_announcements_table.py` is the clean, autogenerated version.
