# 05 ‚Äî mobile: fetch, dismissal tracking, modal, dot badge

**What to build:** The same student-facing behavior as ticket 04, ported to `mobile`: fetch the current announcement, persist `lastSeenAnnouncementId` via AsyncStorage, and surface it as an auto-shown modal, a settings-gear dot badge, and a permanent settings-sheet entry.

**Blocked by:** 02

**Status:** ready-for-agent

- [x] `mobile/src/lib/storage.ts` gains `getLastSeenAnnouncementId()`/`setLastSeenAnnouncementId(id)` over AsyncStorage (key `kiit-time:last-seen-announcement`)
- [x] `mobile/src/lib/announcements.ts` ‚Äî duplicated `isAnnouncementUnseen` from webapp (no shared package exists between clients, matches this repo's existing per-client duplication convention)
- [x] Unit tests added: `lib/__tests__/announcements.test.ts` (4 cases) + 3 new cases appended to the existing `lib/__tests__/storage.test.ts`. Full suite: 47/47 passing (`pnpm test`)
- [x] `mobile/src/hooks/useAnnouncement.ts` wraps `fetchCurrentAnnouncement` in TanStack Query, key `['announcement/current']`, `staleTime: 0`
- [x] `mobile/src/components/announcement-dialog.tsx`, built the same way as `about-dialog.tsx`, shows title, body, tappable link when present. **Note**: `mobile-ui-polish/03` (About dialog full-width fix) has **not** landed yet ‚Äî `about-dialog.tsx` still has the broken no-op `pr-15 pl-15` classes. `announcement-dialog.tsx` intentionally does not copy that broken pattern, but both dialogs share the same cramped-width underlying issue until `mobile-ui-polish/03` is actually fixed.
- [x] On app open, an unseen active announcement auto-opens the dialog. Implementation note: initially wrote this as a `useEffect` calling `setState`, which this repo's `react-hooks/set-state-in-effect` ESLint rule correctly flagged (cascading-render anti-pattern) ‚Äî rewrote using React's recommended "adjust state during render" pattern instead (an `autoOpenedForId` state variable compared inline during render, not in an effect). `eslint` and `tsc --noEmit` both clean (the only `tsc` errors are pre-existing, in untouched `ui/input.tsx`/`ui/textarea.tsx`).
- [x] The settings-gear FAB shows a dot badge (`bg-brand`) when unseen, clearing on dismissal via the same derivation. (RN `View`'s default `position: relative` meant no `fixed`-vs-`relative` conflict like the one hit and fixed on webapp.)
- [x] `mobile/src/components/settings-sheet.tsx` gains an optional `onAnnouncementPress` prop rendering a permanent "Announcement" row (same style as Contact/About), only shown when provided (i.e. when an announcement exists) ‚Äî reopens the dialog regardless of seen state. **Deviation from spec**, same as webapp: marks it seen on *close*, not on *open* ‚Äî the shared `onOpenChange` handler covers both entry paths since closing is the only way to leave the dialog.
- [x] Manual verification via `pnpm expo` ‚Äî done; caught two real issues during testing, both fixed:
  1. **Stale announcement after admin clears it**: `mobile/src/lib/query-client.tsx`'s `QueryProvider` never wired React Native's `AppState` to TanStack Query's `focusManager`. Query's refetch-on-focus relies on browser window-focus events by default, which don't exist in RN ‚Äî so `staleTime: 0` queries (the announcement) fetched once and then never refetched, even across backgrounding/foregrounding the app. This affected every query app-wide, not just announcements. Fixed by listening to `AppState` changes and calling `focusManager.setFocused()`, per TanStack Query's documented React Native integration pattern.
  2. **Dismiss-marks-seen was too aggressive** (user feedback, applies to webapp too ‚Äî see ticket 04 amendment): closing the dialog via the X button or tapping outside also permanently marked the announcement as seen, clearing the notification dot. Since a tap-outside dismissal can be accidental, this meant a user could lose the "there's something new" signal without ever reading it, with no way to get it back. Changed to a "Mark as read" pattern: `onOpenChange` now only controls dialog visibility; a new explicit "Mark as read" button inside `announcement-dialog.tsx` (and `AnnouncementDialog.tsx` on webapp) is the only thing that calls `setLastSeenAnnouncementId`. Closing without pressing it leaves the announcement unseen ‚Äî dot badge and settings-sheet auto-reopen behavior both persist until explicitly acknowledged.
  3. **Follow-up polish** (user feedback): the settings-sheet "Announcement" row now also shows its own small unseen dot (`announcementUnseen` prop threaded through `SettingsSheet`), not just the FAB ‚Äî visible even once the sheet is open. The "Mark as read" button now only renders when `unseen` is true (hidden once already read, avoiding a misleading always-present CTA). Restyled from a solid `bg-brand` fill to the same muted `bg-surface border border-border` secondary style used elsewhere in the settings sheet ‚Äî per button-hierarchy UX guidance (Material Design emphasis levels; DesignMonks/Cieden button guides), a low-stakes acknowledgment action shouldn't compete visually with primary actions via a vibrant fill.
  5. **Settings sheet closed on "Announcement" tap, unlike "About"** (user feedback): the mobile `onAnnouncementPress` handler called `settingsSheetRef.current?.dismiss()` before opening the dialog, while `onAboutPress` doesn't touch the sheet at all ‚Äî an inconsistency. Removed the `dismiss()` call so both entries behave identically. Found and fixed the same inconsistency on webapp too (its "Announcement" button called `setSheetOpen(false)`, "About" didn't) for parity across clients.
  4. **Mobile-only race condition**: after marking read and reloading the app, the dialog popped back open with the "Mark as read" button already missing ‚Äî looked like the dot/button "weren't implemented," but the real bug was that `AsyncStorage.getItem` (used for `getLastSeenAnnouncementId`) is async, while the auto-open check runs synchronously during the first render. On that first render `lastSeenAnnouncementId` was still its initial `null` (storage hadn't resolved yet), so the unseen check evaluated `true` against stale data and wrongly auto-opened the dialog ‚Äî then, moments later, the real persisted value loaded and corrected `announcementUnseen` to `false`, hiding the dot/button after the dialog had already (wrongly) opened. Fixed by gating both the auto-open check and the `announcementUnseen` derivation on a new `lastSeenLoaded` flag that only flips true once the AsyncStorage read actually resolves. webapp doesn't have this bug ‚Äî `localStorage` is synchronous, so its equivalent lazy `useState` initializer is already correct on the first render.
